# Zo Space

This directory powers the user's zo.space site. **DO NOT edit files here directly** — they are generated by the space tools and your changes will be overwritten.

This README documents the mounted default-space shell for humans and agents
inspecting `/__substrate/space` directly. Normal route-writing agents mostly see
the zo.space guidance in `src/hostagent/prompts/sysprompt_parts.py` and tool
docs, not this file, so put route-authoring rules there first and keep this file
as supporting shell documentation.

## How to manage your space

Use the space tools in chat:

- `write_space_route(path, route_type, code, public)` — Create or fully rewrite a route
- `edit_space_route(path, code_edit, edit_instructions, public)` — Patch an existing route
- `delete_space_route(path)` — Remove a route
- `list_space_routes()` — List all routes (returns IDs and paths, not code)
- `get_space_route(route_id)` — Get a route's full details including code
- `update_space_asset(source_file, asset_path)` — Copy a workspace file to assets
- `delete_space_asset(asset_path)` — Delete an asset
- `list_space_assets()` — List all uploaded assets
- `get_space_errors()` — Check for runtime errors in routes

Or manage routes visually in the **Space** tab.

## Shell vs starter homepage

`src/App.tsx` is the app shell for every Space route. Keep cross-route behavior
there: route mounting, SEO defaults, error boundaries, and persistent
attribution.

`src/pages/Home.tsx` is starter/fallback homepage content. Its theme picker,
prompt buttons, example cards, and local helper components are onboarding
scaffolding for new Spaces. Generated routes do not automatically inherit those
helpers; if a generated page needs a component, define it in that route or build
it from semantic JSX/Tailwind.

The starter homepage stores its theme/background preset values directly in
`Home.tsx`, not in hidden shell state. This makes it easy for an agent to tweak a
homepage theme by editing one local object. The homepage picker is only a local
preview for the current browser session; it does not persist as a website
change. Generated pages should use the same local-object pattern when they need
page-specific theming.

Global CSS variables in `src/styles.css` are a baseline, not a constraint.
Generated pages should usually use semantic theme classes such as
`bg-background`, `text-foreground`, `bg-card`, and `text-muted-foreground`, but
page-specific art direction can define local styles inside that route.

## Dependencies and URL imports

Zo Space has a fixed set of pre-installed dependencies. Route-writing agents
should not run `bun add`, `npm install`, `pnpm add`, or edit `package.json`.

For frontend page routes, Vite can load browser-safe ESM modules from full HTTPS
URLs. Prefer pinned `esm.sh` URLs when a page needs a frontend-only package that
is not already installed, for example:

```ts
import confetti from "https://esm.sh/canvas-confetti@1.9.3";
```

Keep core libraries such as React on the installed package imports. API routes
run in the Bun/Hono server, not Vite's browser bundle, so do not use HTTPS URL
ESM imports there. If a project needs server-side packages, custom dependency
management, or build/config control, create a Zo Site instead.

## Icon customization

Default Space's shell uses Hugeicons for UI/system icons via
`src/components/icons.tsx`. Use that file as the central place to add or swap
icons used by the shell itself.

- **Hugeicons** (`@hugeicons/core-free-icons`) is installed by default. It has a polished outline style and a broad catalog: https://hugeicons.com/icons
- **Lucide** (`lucide-react`) is the safest default for generated Space routes and older Spaces. It has simple geometric line icons: https://lucide.dev/icons
- **Tabler** (`@tabler/icons-react`) remains installed for compatibility with generated routes that already use it.
- **Phosphor** (`@phosphor-icons/react`) is a good option for friendly icons with multiple weights/fills: https://phosphoricons.com
- **Brand logos** should usually be exact SVG paths or brand icon packages, not generic UI icons.

Do not remove existing dependencies from `package.json` just because the current
shell does not import them. Many existing Spaces and generated routes may still
depend on packages that were available when they were created. For shell
maintainers, adding a new baked dependency is possible, but existing Spaces may
not receive it without a migration.

## Built on Zo attribution

The default shell includes a fixed `BuiltOnZoBadge` in `src/App.tsx`. Treat it
as persistent attribution: keep it when customizing or redesigning the Space.
You may restyle it or make it more subtle, but make sure it is still present on
the page even if small, and try your best not to remove, hide, or replace it
during unrelated edits. Remove it only when the user explicitly asks to remove
the Zo attribution / "Built on Zo" badge.

If the user explicitly asks to hide the badge on a generated page, do not edit
the app shell. Add route-local CSS that targets `[data-zo-built-on-badge]`.
